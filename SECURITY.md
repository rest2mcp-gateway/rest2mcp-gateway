# Security Policy

## Supported Versions

This project is still in an early stage. Security fixes will be made against the current `main` branch and the most recent published release, if one exists.

## Reporting a Vulnerability

Please do not open public GitHub issues for security vulnerabilities.

Instead, report suspected vulnerabilities privately to the maintainer with:

- a clear description of the issue
- affected components or endpoints
- reproduction steps or a proof of concept
- impact assessment if known
- any suggested mitigation

If GitHub private vulnerability reporting is enabled for this repository, use that channel. Otherwise, contact the maintainer directly through the project contact method listed in the repository profile or documentation.

## What to Expect

When a valid report is received, the maintainer will try to:

- acknowledge receipt in a reasonable time
- reproduce and assess the issue
- prepare a fix or mitigation
- coordinate disclosure once users have a reasonable path to update

Response times are best effort and may vary depending on maintainer availability.

## Scope

Security reports are especially helpful for issues involving:

- authentication and authorization
- token validation
- secret handling and encryption
- access control around runtime publishing and execution
- request forgery, injection, or privilege escalation
- insecure default configuration

## Disclosure Guidance

Please avoid public disclosure until the issue has been reviewed and a fix or mitigation has been prepared.
